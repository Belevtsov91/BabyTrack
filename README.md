## 🧬 История создания

Всё началось с одного приложения из **App Store** — простого, но невероятно полезного трекера для малышей. Оно реально помогало в первые два года: каждое кормление, каждый сон, каждый подгузник — всё под рукой в любое время ночи.

Но у него были свои ограничения. И однажды пришла мысль: *а что если взять эту идею и сделать её лучше? Добавить то чего не хватало, убрать то что раздражало, и построить что-то своё.*

Так **25 января 2026 года** появился этот проект. И это не финальная точка — это только начало. Приложение будет постоянно развиваться, улучшаться и дополняться новыми фичами. Потому что хорошие идеи не заканчиваются.

---

Этот проект — честный ответ на вопрос **"а что такое вайб-кодинг на самом деле?"**

```
1. 💡 Идея             →  Lovable.dev       (накидал UI за вечер, магия)
2. 🔧 Доработка        →  Claude + ChatGPT  (объяснял, чинил, переписывал)
3. 🚀 Финальный вид    →  Руки + голова     (куда же без этого)
```

**Спойлер:** вайб-кодинг работает. Но не так как на TikTok показывают.

> *Без минимального понимания что происходит в коде — ты не вайб-кодишь,*
> *ты просто жмёшь кнопки и молишься. Это называется иначе.*

Разница между "вайб-кодером" и "просто надеющимся":
- Ты понимаешь **что** просишь сделать, даже если не знаешь **как**
- Ты читаешь что AI написал и понимаешь хотя бы логику
- Ты можешь объяснить баг словами, а не просто "оно не работает"

**Lovable** дал красивый старт за несколько часов вместо нескольких дней.
**Claude** помог разобраться с архитектурой, починить баги и не сломать то что работает.
**ChatGPT** помог сформулировать идеи когда мысли в голове ещё не стали словами.

Итог: проект который реально работает, а не просто скриншот для соцсетей. 🎯

## 🎓 Честно про вайб-кодинг

Biggest lesson этого проекта был не в том, что AI "пишет код за тебя", а в том, что он очень быстро генерирует направление — но почти никогда не доводит продуктовую интеграцию до конца без ручной инженерной работы. Самые красивые куски кода чаще всего ломались не на уровне идеи, а на стыке реального проекта: роутинг, состояние, адаптив, существующие компоненты, локальные данные и уже принятые UI-паттерны.

Каждый инструмент играл свою роль:

> **Lovable** дал визуальный скаффолд за вечер — то, на что вручную ушли бы дни.
> **Claude** помогал с архитектурой, дебагом и связкой компонентов с реальными данными.
> **ChatGPT** — когда мысль ещё не оформилась в конкретный вопрос, помогал нащупать направление.

### С чем реально сталкивались

- AI регулярно приносил красивые компоненты, но под другой стек: `next/link`, странные токены, демо-данные которых у нас нет
- Многие блоки выглядели нормально на скрине, но разваливались на tablet/mobile — особенно sticky-секции, onboarding и сложные hero-блоки
- Часто появлялось дублирование логики: новый красивый компонент сверху, старый рабочий снизу. Потом приходилось вручную убирать второй источник правды
- Эффекты вроде liquid/glow сначала выглядели эффектно, но мешали UX. Пришлось урезать их до hover/focus/active, а не держать интерфейс "светящимся" постоянно
- AI нередко ошибался в деталях TypeScript и архитектуры: конфликты имён файлов, несовместимые пропсы, дубли UI-компонентов, неверные предположения о данных
- Каждую новую сессию нужно было заново объяснять контекст: вот наш auth-flow, вот как устроен storage, вот почему мы не используем X. Без этого AI с уверенностью генерировал решения, конфликтующие с уже принятыми
- AI создаёт красивую абстракцию под свою модель данных, а не под твою. И ты оказываешься перед выбором: переделать данные под компонент или компонент под данные. Оба варианта стоят времени, которого "вайб-кодинг" якобы экономит
- Самая частая реальная работа — не "написать с нуля", а аккуратно связать готовый кусок с нашими events, localStorage, auth-flow и маршрутами

### Что реально помогло

- Использовать AI как быстрый черновик и генератор направлений, а не как финального автора решения
- Сразу проверять как компонент живёт внутри существующего UX, а не оценивать его по изолированному скриншоту
- Быстро резать лишнее: если новый блок создаёт дублирование — старую версию надо либо удалить, либо не внедрять новую
- После каждого заметного изменения прогонять сборку и смотреть реальные разрешения, а не верить "красивому" коду на слово

> *Если коротко: вайб-кодинг сжимает недели в дни — но не так как показывают в TikTok.
> Там показывают первые 20%. Оставшиеся 80% — это обычная инженерная работа, просто с лучшим черновиком в руках.*

## 🛠️ Паттерны и скилы прокачанные в процессе

### React / архитектура

- **`react-hooks/purity`** — `Date.now()` и side effects нельзя вызывать в render. Все event handlers обёрнуты в `useCallback`
- **Functional setState** — избавились от stale closures в таймерах: `setState(prev => ...)` вместо прямого чтения state в колбэке
- **Убрали `AnimatePresence mode="wait"`** из page transitions — exit-анимация блокировала монтирование новой страницы, навигация зависала. Решение: только `motion.div` с enter-анимацией
- **TypeScript без `as any`** — типизация через `BabyEvent['type']` вместо кастов

### UI / дизайн-система

- **`LiquidSurface`** как единственный UI-примитив для всех интерактивных элементов — кнопок, инпутов, карточек
- **`hsl(var(--primary))`** цветовая система через CSS-переменные — никаких хардкодных цветов в коде
- **Категорийные токены** `--sleep`, `--feeding`, `--diaper` — каждый тип события имеет свой цвет из темы
- **Liquid/glow только на hover/focus/active** — не держать интерфейс "светящимся" постоянно, это мешает UX

### Данные / auth

- **localStorage-first архитектура** — все данные локально, полная независимость от backend
- **Полный auth-flow**: регистрация → логин → forgot password → token → session — всё через localStorage
- **btoa-based hashing** для паролей до появления нормального backend

### Инструменты / процесс

- **CLAUDE.md** с правилами principal-level архитектора — read-only по умолчанию, минимальные изменения, cross-layer inspection перед любым решением
- **Memory система** — architectural decisions сохраняются между сессиями чтобы AI не забывал принятые решения
- **MCP серверы**: Context7 для документации, Playwright для автоматических скриншотов, n8n для автоматизации
- **VS Code** настроен под проект: ESLint + Prettier + Tailwind IntelliSense + Path aliases `@/`
